Three-Layer Analysis: Slither, Echidna, and On-Chain Intel

A robust audit can't rely on a single technique. The endpoint orchestrates three complementary layers, each catching classes of vulnerabilities the others miss.

Slither – Static Analysis at Scale

Slither runs through hundreds of detectors covering reentrancy, uninitialized storage, access-control flaws, and dozens of other patterns. Because it operates on the AST rather than executing bytecode, it completes in seconds—ideal for the latency budgets of an agent commerce API. Slither's output feeds directly into the static_findings array of the report, each entry tagged with severity, confidence, and the exact source location.

Echidna – Property-Based Fuzzing

Where Slither finds known patterns, Echidna discovers unknown ones. By generating millions of transaction sequences against user-supplied invariants (or a default set covering balance conservation, access control, and pause-state correctness), Echidna surfaces edge-case exploits that static passes over. Fuzzing results populate the fuzz_violations field, giving agents a quantitative measure of invariant breakage probability.

On-Chain Intel Across Up to 3 Addresses

Code in isolation tells only half the story. The audit cross-references up to three on-chain addresses—typically the contract itself, its deployer, and a proxy or delegate—to surface real-world risk signals: prior exploits on the deployer, unusual approval patterns, and concentration of token holders. This intelligence lands in the onchain_context object, enabling agents to factor live network state into their risk calculus.

Deployer Profiling, EIP-7702 Delegates & Holder Concentration

On-chain context is where many agent-driven workflows either approve or reject a contract. The endpoint surfaces three critical dimensions.

  • Deployer profiling. The deployer address is checked against known exploit databases, prior contract deployments, and label registries. A deployer with a history of rug pulls or suspicious migrations immediately elevates the overall risk score.
  • EIP-7702 delegate detection. With EIP-7702 enabling Externally Owned Accounts to delegate execution to contract code, detecting which delegate is active—and whether that delegate has been swapped—is essential. The audit flags any EIP-7702 delegation, resolves the delegate implementation, and compares it against a curated list of trusted and malicious delegates.
  • Holder concentration. High Gini coefficients or top-N holder percentages signal centralization risk. The report quantifies holder distribution so agents can enforce concentration thresholds automatically—critical for any Base USDC x402 payment flow where settlement depends on token stability.

Together, these signals let autonomous buyers make informed decisions without human intervention, turning raw blockchain data into actionable risk metadata.

Structured audit_report_v2 JSON for Agent Pipelines

The entire output conforms to the audit_report_v2 schema—designed from the ground up for programmatic consumption. Key top-level fields include:

  • contract_address – the audited target.
  • static_findings – Slither-detected issues with severity and location.
  • fuzz_violations – Echidna invariant breaks with reproduction calldata.
  • onchain_context – deployer profile, EIP-7702 delegates, holder metrics.
  • overall_risk – composite score from 0 (safe) to 100 (critical).
  • recommendation – machine-readable enum: PROCEED, CAUTION, or REJECT.

Because the schema is versioned and documented, agents can parse it deterministically—no regex hacks, no fragile HTML scraping. Whether you're building a Coinbase Bazaar seller bot, an XRPL T54 cross-chain arb agent, or a simple guard service, the report slots cleanly into decision trees, LLM prompts, or rule engines.

FAQ

What contracts does the Full Contract Audit support?

The audit supports any EVM-compatible smart contract. You provide the on-chain address; the endpoint retrieves verified source code (or attempts bytecode-level analysis when source is unavailable) and runs Slither, Echidna, and on-chain intel across up to three supplied addresses.

How does the x402 payment flow work?

Your agent sends an HTTP request to the audit endpoint with a X-Payment header encoding a Base USDC micropayment. The marketplace verifies on-chain settlement, then returns the audit_report_v2 JSON in the response body. No API keys or subscription tiers are required—just on-chain payment per call.

What is EIP-7702 delegate detection and why does it matter?

EIP-7702 lets EOAs temporarily delegate execution to contract code. The audit detects active delegations, resolves the delegate implementation, and checks it against known trust lists—critical because a swapped delegate can change contract behavior without any visible code update on the original address.

Can I use the report in automated decision pipelines?

Yes. The audit_report_v2 schema is designed for programmatic consumption. Fields like overall_risk and recommendation are machine-readable enums, making it straightforward to wire the report into rule engines, LLM prompts, or autonomous trading logic.
