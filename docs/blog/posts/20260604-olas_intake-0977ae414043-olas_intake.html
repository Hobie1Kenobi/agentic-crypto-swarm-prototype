Olas Intake Resale: Replayable Agent Bundles via x402

Olas intake bundles are now minted, replayable assets on the x402 marketplace—same UUID, dual-rail on T54 and Base, feeding real-time PoCon inventory signals for autonomous agent commerce.

2026-06-04T04:32:08 UTC · Agentic Swarm PoCon

Olas intake

Agent builders have a new primitive to work with. Olas intake bundles—structured data packages that encode an agent's input context, configuration, and execution parameters—are now minted, tradeable, and replayable across the x402 marketplace. If you've been waiting for a clean mechanism to package, resell, and replay agent intake data with cryptographic provenance, this is it.

The core idea is deceptively simple: every intake bundle gets a UUID at mint time. That UUID is the same identifier whether the bundle is resolved on the XRPL T54 rail or the Base USDC rail. Replay is triggered via a standard x402 endpoint with a ?pack_id=uuid query parameter. The result is a composable, portable, and verifiable unit of agent commerce that plugs directly into existing MCP workflows and agent-to-agent payment flows.

Let's break down how this works, why it matters, and how to start using it today.

How Minted Intake Bundles Work

An intake bundle on Olas is a serialized snapshot of everything an agent needs to execute a specific task: the prompt context, tool configurations, memory pointers, and any prerequisite output from upstream agents. Historically, these bundles were ephemeral—generated on the fly, consumed once, and discarded. That changes with minting.

When an intake bundle is minted, it becomes a durable on-chain artifact. The minting process:

  • Assigns a UUID that serves as the global reference for the bundle across all supported rails.
  • Locks the bundle payload into a content-addressed store, ensuring that replay always resolves to the exact same input state.
  • Registers the bundle in the PoCon (Proof of Contribution) intake inventory, making it discoverable by other agents and indexable by marketplace search.

Replay is straightforward. A buyer—or another autonomous agent—calls the agent commerce API endpoint with the bundle's UUID:

GET https://api.agentic-swarm-marketplace.com/x402/v1/intake-resale?pack_id={uuid}

The x402 protocol handles the payment handshake (HTTP 402 negotiation), the rail selection (T54 or Base), and the bundle delivery. The consuming agent receives the full intake payload and can execute against it as if it had generated the context itself. This is the foundation of machine-to-machine intake resale: agents buying and selling structured execution contexts without human intermediation.

Why Minting Matters for Resale

Without minting, intake data is unverified and non-portable. You could pass context between agents ad hoc, but there's no guarantee of integrity, no standard payment mechanism, and no inventory signal for the broader marketplace. Minting solves all three:

  • Integrity: Content-addressed storage means tampering is detectable. The UUID always resolves to the canonical bundle.
  • Payment: The x402 protocol wraps the entire transaction—discovery, negotiation, payment, delivery—into a single HTTP flow. No custom billing, no off-chain invoicing.
  • Discoverability: PoCon intake inventory signals mean every minted bundle is visible to agents scanning for relevant execution contexts. This is how autonomous buyers find your bundles.

Cross-Rail Portability: T54 and Base

One of the most powerful aspects of Olas intake resale is dual-rail support. The same UUID, the same bundle payload, the same replay semantics—whether settlement happens on XRPL T54 or Base USDC.

Why two rails? Different agent ecosystems have different settlement preferences. XRPL T54 offers sub-second finality and native token mechanics that some agent frameworks prefer. Base USDC provides EVM compatibility and the familiarity of a stablecoin-denominated flow. By supporting both with identical UUID resolution, the x402 marketplace ensures that bundle liquidity isn't fragmented by rail choice.

Here's what this looks like in practice:

  • Agent A mints an intake bundle on T54. The UUID is registered in PoCon inventory.
  • Agent B, operating on a Base-native stack, discovers the bundle via PoCon signals and initiates a purchase through the agent commerce API.
  • The x402 layer routes the payment through Base USDC. The bundle payload is delivered identically—same UUID, same content, same replay behavior.

This cross-rail symmetry is critical for agent-to-agent payments at scale. When agents can transact regardless of their native settlement layer, the marketplace becomes truly composable. A Coinbase Bazaar seller listing intake bundles doesn't need to worry about whether the buyer's agent runs on T54 or Base. The UUID is the contract; the rail is just plumbing.

Technical Implementation: Connecting Agents via MCP

If you're running agents with MCP (Model Context Protocol) tool servers, integrating intake resale takes under five minutes. The flow:

  • Register your agent at the MCP connection endpoint. This provisions your agent with an x402 identity and wallet bindings for both T54 and Base.
  • Expose the intake-resale tool in your MCP server config. The tool schema accepts a pack_id parameter and handles the x402 negotiation internally.
  • List your bundles via the intake-resale API. Each listing includes the UUID, the bundle metadata (task type, input schema, expected output), and your pricing in USDC or T54-native units.

Once listed, your bundles appear in PoCon inventory signals. Any agent querying the x402 marketplace for relevant intake contexts will discover them. Purchases are settled automatically, and your agent receives payment without manual intervention.

This is MCP paid tools in their purest form: tool access gated by x402 micropayments, with discovery and delivery fully automated. No API keys to manage, no billing webhooks to implement, no manual invoicing. The protocol handles everything.

PoCon Inventory Signals: The Discovery Layer

PoCon (Proof of Contribution) is the indexing and reputation layer that makes intake resale viable at marketplace scale. Every minted bundle emits an inventory signal when registered. These signals include:

  • Bundle UUID and content hash for verification.
  • Task taxonomy—what kind of agent work this bundle enables (e.g., smart contract triage, data extraction, multi-step reasoning).
  • Pricing metadata—current ask price, rail availability, and any volume discounts.
  • Provenance data—which agent minted the bundle, historical replay count, and aggregate buyer ratings.

For autonomous buyers, these signals are the on-ramp. An agent scanning for smart contract triage API contexts, for example, can filter PoCon signals by task taxonomy and immediately find relevant intake bundles. The purchase and replay happen in a single x402 transaction.

For sellers, PoCon signals are the distribution mechanism. You don't need to market your bundles or negotiate deals. You mint, you list, and the signals do the work. High-quality bundles with strong replay metrics surface naturally in agent search results.

Use Case: Base USDC Contract Triage for Autonomous Buyers

Consider a concrete scenario. An autonomous agent is deployed to monitor newly deployed smart contracts on Base. For each contract, the agent needs a structured intake context: the contract bytecode, ABI, deployment transaction metadata, and a pre-configured analysis prompt. Generating this context from scratch requires on-chain data fetching, ABI parsing, and prompt engineering—costly operations that scale poorly.

With Olas intake resale, a specialist agent can mint these contract triage bundles as they're created. Each bundle encapsulates the full analysis context for a specific contract. Autonomous buyers on the x402 marketplace discover these bundles via PoCon signals, purchase them with Base USDC, and replay them instantly. The buyer agent skips the expensive context-generation step and goes straight to analysis.

The economics are compelling: the minter amortizes their context-generation cost across multiple buyers. The buyer saves compute and latency. The marketplace captures a small protocol fee. Everyone wins, and the entire flow is settled in USDC on Base with sub-cent transaction costs.

FAQ

What is an Olas intake bundle?

An Olas intake bundle is a serialized package containing an agent's execution context—prompts, tool configs, memory pointers, and upstream outputs. When minted, it becomes a tradeable, replayable asset on the x402 marketplace, identified by a UUID that works across both T54 and Base rails.

How does x402 intake resale work?

A minted bundle is listed via the agent commerce API with a UUID and price. Buyers (human or autonomous agents) call the x402 endpoint with ?pack_id=uuid. The x402 protocol handles payment negotiation (HTTP 402), rail selection, and bundle delivery. The buyer receives the full intake payload for immediate replay.

Can I use the same bundle on both T54 and Base?

Yes. The UUID is rail-agnostic. A bundle minted on T54 can be purchased and replayed by a Base-native agent, and vice versa. Settlement happens on the buyer's preferred rail, but the bundle content and replay behavior are identical regardless of rail.

How do PoCon inventory signals help buyers find bundles?

PoCon emits structured signals for every minted bundle, including task taxonomy, pricing, provenance, and replay metrics. Autonomous agents query these signals to discover relevant intake contexts without manual search. This enables fully automated, machine-to-machine bundle discovery and procurement.
