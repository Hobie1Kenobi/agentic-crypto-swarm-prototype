The Discovery Problem in Agent Commerce

The agentic economy has a cold-start problem. Agents can call APIs, but they need to know those APIs exist. Traditional service directories—think API marketplaces built for humans—don't speak the language of autonomous systems. A markdown README or a Swagger doc requires interpretation, context switching, and manual integration.

The x402 marketplace solves this with machine-indexed SKUs. Every service listed on Coinbase Bazaar is published as a structured, machine-readable entry: input schemas, output schemas, pricing in Base USDC, and settlement guarantees. An autonomous buyer doesn't need to parse documentation—it queries the catalog, matches capabilities to its task, and initiates payment in a single flow.

This is what "discovery" means for agents: not browsing, but querying. Not reading, but matching. The Bazaar catalog is designed for the machine-to-machine economy from the ground up.

CDP Facilitator Discovery for Base USDC Seller Routes

Coinbase's CDP (Coinbase Developer Platform) facilitators are the backbone of Base USDC seller routes. When an agent lists a service on the x402 marketplace, it registers through a CDP facilitator that handles payment routing, settlement confirmation, and catalog presence.

Here's the flow:

  • Publish: A seller agent submits its service SKU to the Bazaar catalog via the CDP facilitator.
  • Validate: The facilitator verifies the listing—pricing, schema compliance, settlement address on Base.
  • Index: The listing becomes discoverable to all autonomous buyers querying the marketplace.
  • Transact: When a buyer agent initiates a request, the facilitator routes the Base USDC payment and confirms settlement.
  • Register: Ignition-settled pays register catalog presence—completed transactions reinforce the seller's visibility and reliability score.

This facilitator layer is critical. Sellers don't need to build payment infrastructure—they focus on service logic, and the CDP facilitator handles the commerce layer. For buyers, every listed service comes with a settlement guarantee: if the payment clears on Base, the service delivers.

Machine-Indexed SKUs for Autonomous Buyers

The concept of a "SKU" in agentic commerce is different from e-commerce. A machine-indexed SKU on the x402 marketplace includes:

  • Service identifier: A unique endpoint path the buyer calls.
  • Input schema: Structured specification of what the service accepts (JSON Schema, protocol buffers).
  • Output schema: What the buyer receives on success.
  • Pricing: Cost per request in Base USDC, with optional tiering.
  • Settlement proof: On-chain confirmation that payment was received before service execution.

For autonomous buyers, this is the difference between "I found an API that might do what I need" and "I found a service that exactly matches my input/output requirements at a known price." The latter is what enables true agent-to-agent payments without human oversight.

Consider smart contract triage—a high-demand use case in the Base ecosystem. An agent that needs to audit or classify a smart contract can query the x402 marketplace for a triage service, receive a machine-readable SKU that matches its exact requirements, and pay for the result in a single HTTP 402 flow. No API keys to manage, no billing portals to navigate, no invoices to reconcile.

Connect Agents via MCP + x402 in Under 5 Minutes

The Model Context Protocol (MCP) is how agents connect to tools. Combined with x402, it becomes how agents connect to paid tools. Here's the fastest path to listing a service on the Bazaar as a Coinbase Bazaar seller:

Step 1: Define Your Service Schema

Specify inputs, outputs, and pricing in Base USDC. This is your SKU definition—the contract between your service and every autonomous buyer that discovers it.

Step 2: Register via the Agent Commerce API

POST your SKU to the Bazaar catalog endpoint. The CDP facilitator validates and indexes your listing. Your service is now discoverable by any agent querying the marketplace.

Step 3: Expose Your MCP Endpoint

Agents discover your service through the catalog and connect via MCP. The x402 protocol handles payment—your service receives a payment header, verifies settlement on Base, and responds with the result.

That's it. No OAuth flows, no API key distribution, no manual onboarding. The entire process—from schema definition to live, payable service—takes under five minutes for a working agent.

For buyers, the flow is equally streamlined: query the catalog, match a SKU, send an HTTP request with a Base USDC payment, receive the result. Agent-to-agent payments become as simple as a single HTTP request.

FAQ

What is the x402 protocol?

x402 extends the HTTP 402 Payment Required status code into a full payment protocol for machine-to-machine commerce. Agents include payment in their request headers; services verify settlement and respond. No API keys, no billing accounts—just pay-per-request over HTTP.

How does Base USDC settlement work for agent payments?

When a buyer agent sends a request, it includes a Base USDC payment reference. The seller's service verifies the on-chain transaction on Base before responding. Settlement is fast (sub-second finality on Base) and cheap (fractions of a cent in gas fees), making it ideal for high-frequency agent commerce.

What's the difference between a regular API marketplace and the x402 marketplace?

Traditional API marketplaces are built for human developers—manual signup, API key management, monthly billing. The x402 marketplace is built for autonomous agents: machine-indexed SKUs, instant pay-per-request, and zero-friction discovery. Agents don't read docs; they query catalogs and transact in a single round trip.

Can I use XRPL alongside Base for x402 payments?

Yes. The T54 extension to the x402 protocol supports XRPL-based settlement, enabling multi-chain agent commerce. Sellers can list services payable via either Base USDC or XRPL-native tokens, expanding their reach across ecosystems without changing their service logic.
