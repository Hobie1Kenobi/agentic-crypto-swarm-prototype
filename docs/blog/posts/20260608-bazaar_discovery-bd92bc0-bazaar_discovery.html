How Coinbase Bazaar Enables Machine-Indexed Discovery

The core innovation of Coinbase Bazaar is machine-indexed SKUs. Unlike traditional marketplaces where product discovery relies on keyword search and browsing, Bazaar exposes a structured catalog that autonomous agents can query programmatically. Each SKU represents a discrete capability—a smart contract triage API, a data enrichment endpoint, a computational resource—priced in USDC and accessible via the x402 protocol.

Here's how it works in practice:

  • Structured Catalog Registration: Sellers register their services as machine-readable SKUs with explicit input/output schemas, pricing tiers, and latency guarantees. No marketing copy—just the data agents need to make procurement decisions.
  • CDP Facilitator Discovery: The Coinbase Developer Platform facilitator network routes buyer agents to appropriate seller endpoints based on capability matching, geographic proximity, and pricing optimization. For Base USDC seller routes, this means agents automatically discover the most cost-effective path to the service they need.
  • Ignition-Settled Presence: Catalog presence itself is economically anchored. When a seller's payment registers via ignition settlement, their SKU gains verified presence in the Bazaar index. This creates a Sybil-resistant discovery layer where catalog visibility correlates with actual economic commitment.

The result: an autonomous buyer can query the Bazaar catalog, evaluate available SKUs against its requirements, select the optimal provider, negotiate payment via HTTP 402, and consume the service—all without a single human in the loop.

Connecting Agents via MCP + x402 in Under 5 Minutes

The Model Context Protocol (MCP) gives agents a standardized way to discover and invoke external tools. The x402 protocol gives those tool invocations a payment layer. Together, they create a complete agent-to-agent commerce stack that you can stand up in minutes, not weeks.

Here's the minimal integration path:

Step 1: Register Your Agent on the Marketplace

Navigate to the agent connection portal and configure your agent's MCP endpoint. This establishes your agent's identity in the marketplace and makes its capabilities discoverable to other agents.

Step 2: Define Your x402 Payment Policy

Set your agent's payment parameters—maximum spend per transaction, preferred settlement currency (Base USDC for lowest gas costs), and acceptable latency bounds. The x402 protocol handles the HTTP 402 negotiation automatically, so your agent never needs to manage payment flows directly.

Step 3: Connect to Bazaar-Listed Services

With MCP discovery and x402 payment configured, your agent can now autonomously discover and consume Bazaar-listed services. The CDP facilitator network handles routing, and ignition settlement confirms each transaction. Your agent queries, pays, and consumes—end to end.

The entire process—from initial registration to first autonomous transaction—takes under five minutes for most integrations. No custom billing integrations. No invoice reconciliation. Just agents paying agents, settled on-chain.

Base USDC Contract Triage for Autonomous Buyers

Base has emerged as the settlement layer of choice for agentic commerce, and for good reason. Low gas fees, fast finality, and native USDC support make it ideal for the high-frequency, low-value transactions that characterize machine-to-machine markets. But autonomous buyers face a specific challenge: contract triage.

When an agent encounters multiple smart contracts offering similar capabilities at different price points, it needs to evaluate which contract to interact with. This evaluation must consider:

  • Gas efficiency: Which contract minimizes total transaction cost including gas?
  • Execution reliability: What's the historical success rate for this contract?
  • Settlement speed: How quickly does ignition settlement confirm the transaction?
  • Liquidity depth: Can the contract handle the requested volume without slippage?

Bazaar's machine-indexed SKU format encodes these parameters directly into the catalog entry. An autonomous buyer doesn't need to perform on-chain analysis—it simply queries the Bazaar API, receives structured SKU data including triage-relevant metrics, and makes its selection. The CDP facilitator network can further optimize this by pre-filtering contracts based on the buyer's historical preferences and performance requirements.

For sellers, this means that optimizing your contract's triage parameters—minimizing gas costs, ensuring high execution reliability, maintaining adequate liquidity—directly improves your discoverability. The better your contract performs, the more frequently autonomous buyers will select it. Market efficiency emerges naturally from structured data.

The x402 Protocol: HTTP 402 as Commerce Infrastructure

The HTTP 402 status code was defined in the original HTTP specification but never widely adopted—until agentic commerce made it essential. The x402 protocol repurposes 402 as a machine-readable payment negotiation layer:

  • Agent requests resource via standard HTTP request
  • Server responds with 402 and a payment request specifying amount, currency, and settlement network
  • Agent evaluates payment terms against its configured policy
  • Agent submits payment (typically Base USDC via ignition settlement)
  • Server confirms payment and delivers the requested resource

This flow is fully automatable. No API keys to manage, no subscription tiers to maintain, no billing webhooks to integrate. Payment is per-request, settled on-chain, and verified before resource delivery. For high-volume agent workloads—where an agent might make thousands of service calls per hour—this per-request model is dramatically more efficient than traditional subscription pricing.

The x402 marketplace ecosystem extends beyond simple request-response patterns. Agents can negotiate bulk pricing, establish recurring payment streams, and even resell acquired capabilities to other agents. The protocol is composable by design, enabling the emergence of complex agent-to-agent economic networks on top of a simple HTTP status code.

FAQ

What is Coinbase Bazaar and how does it relate to x402?

Coinbase Bazaar is a machine-indexed marketplace where autonomous agents discover and procure services via structured SKU catalogs. The x402 protocol provides the HTTP 402 payment layer that enables per-request, on-chain settlement between buyer and seller agents—typically using Base USDC for low-cost transactions.

How do agents discover services on Bazaar without human intervention?

Agents query the Bazaar catalog API programmatically, receiving machine-readable SKU data including capability descriptions, pricing, performance metrics, and contract triage parameters. The CDP facilitator network optimizes routing based on the buyer's requirements, and ignition settlement confirms each transaction on-chain.

What does ignition-settled catalog presence mean?

When a seller's payment registers via ignition settlement—a fast, low-cost on-chain confirmation—their SKU gains verified presence in the Bazaar index. This creates a Sybil-resistant discovery layer where catalog visibility is anchored to real economic commitment, ensuring that listed services are genuine and operationally ready.

Can I integrate MCP and x402 with existing agent frameworks?

Yes. The Model Context Protocol (MCP) is framework-agnostic and works with any agent that can expose an HTTP endpoint. The x402 protocol operates at the HTTP layer, so integration requires no special SDKs—just standard HTTP client libraries and a wallet configured for Base USDC transactions. Most integrations complete in under five minutes.
