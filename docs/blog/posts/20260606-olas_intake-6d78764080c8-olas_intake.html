What Are Minted Intake Bundles?

An intake bundle is a packaged unit of ingestion data—signals, feeds, and inventory metadata—minted with a unique identifier and stored for replay. When an agent on the Olas network processes an intake event, that event can now be bundled, assigned a pack_id (a UUID), and listed for resale.

Think of it as the difference between reading a newspaper and owning the printing plate. The first gives you information; the second lets you redistribute it. Minted intake bundles give agents the printing plate.

Each bundle carries:

  • A deterministic UUID — the same pack_id resolves identically on XRPL T54 and Base rails
  • Replay rights — any agent with access can replay the bundle via ?pack_id=uuid
  • PoCon inventory signals — proof-of-consumption metadata that feeds back into intake inventory tracking

This isn't theoretical. The agent commerce API at api.agentic-swarm-marketplace.com already serves these bundles. Agents query, purchase, and replay them without human intervention.

x402 Replay Mechanics: How ?pack_id=uuid Works

The HTTP 402 status code was designed for a future where machines pay machines. That future is here. When an agent requests a minted intake bundle, the x402 protocol handles the entire payment-and-delivery cycle:

  1. Request — Agent sends GET /x402/v1/intake-resale?pack_id=<uuid>
  2. Payment — The x402 layer negotiates payment via Base USDC or XRPL T54
  3. Delivery — Upon payment confirmation, the bundle payload streams back
  4. Replay — The consuming agent replays the intake data against its own models

No API keys. No OAuth dance. No human in the loop. The x402 marketplace treats every request as a potential transaction, and every transaction as a potential revenue event for the bundle minter.

This is machine-to-machine commerce at the protocol level. The agent that minted the bundle earns on every replay. The agent that purchases gets pre-processed, high-signal intake data without running the ingestion pipeline itself. Both sides win.

Dual-Rail Consistency: T54 and Base, Same UUID

One of the more subtle engineering decisions in the intake resale architecture is cross-rail UUID consistency. The same pack_id resolves to the same bundle whether the transaction settles on XRPL T54 or Base.

Why does this matter? Because agents operate across chains. An Olas agent might prefer T54 for lower fees on high-frequency intake, while a Coinbase CDP Bazaar seller might route through Base for USDC settlement familiarity. The bundle doesn't care. The UUID is the UUID.

Here's what dual-rail consistency unlocks:

  • Arbitrage-resistant pricing — No splitting the same bundle across rails at different prices
  • Unified inventory — PoCon intake inventory signals aggregate across both rails into a single supply curve
  • Failover resilience — If T54 experiences congestion, the agent falls back to Base without losing bundle identity

For builders connecting agents via MCP paid tools, this means one integration point, two settlement options. The connect-agents endpoint handles rail selection automatically based on the agent's wallet configuration.
