Olas Intake Resale: Replayable Bundles on x402 Rails

Olas intake bundles minted once, replayable everywhere. Same UUID on T54 and Base—agents resell intake inventory signals via x402.

Intake data has always been a first-class asset in agentic commerce—but until now, reselling it meant rebuilding the bundle every time a new agent wanted access. That friction ends today. With Olas intake resale on the x402 marketplace, minted intake bundles become replayable, portable, and monetizable across both T54 and Base rails. One UUID. Two chains. Infinite replay.

Here's what that means for builders: your agents can mint an intake bundle once, attach it to a persistent ?pack_id=uuid, and let any downstream agent pay to replay it via a standard HTTP 402 payment flow. No re-ingestion. No re-processing. The same cryptographic commitment that landed on XRPL T54 validates on Base USDC. This is machine-to-machine commerce at the protocol level.

How Minted Intake Bundles Work

An intake bundle is a structured snapshot of processed signals—think smart contract triage results, on-chain event summaries, or filtered DeFi opportunity feeds. When an agent mints a bundle, it commits the bundle's content hash and metadata to the Olas intake inventory, producing a unique pack_id (a UUID) that permanently identifies that dataset.

The replay mechanism is elegantly simple:

  • Mint once — The producing agent processes raw signals, structures them, and mints the bundle via the agent commerce API.
  • Attach the UUID — The bundle URL becomes /x402/v1/intake-resale?pack_id=uuid. This is the replayable endpoint.
  • Pay to replay — A consuming agent sends an HTTP 402 request with a Base USDC micropayment (or XRPL T54 payment, depending on the rail). Upon verification, the full bundle payload is returned.
  • Same UUID, same data — Whether the consumer hits the endpoint from a Base-native agent or an XRPL T54 workflow, the underlying file is identical. Cross-rail integrity is guaranteed by content-addressed storage keyed to the UUID.

This architecture eliminates the single-use problem that plagued earlier intake systems. Previously, each consumption event required a fresh ingestion pipeline. Now, the producing agent does the work once, and every subsequent replay generates revenue with zero marginal compute. That's the power of agentic commerce: produce once, sell forever.

Cross-Rail Integrity: T54 and Base, Same UUID

One of the most requested features from the agent builder community was cross-chain portability. Intake data shouldn't be siloed to a single rail. With Olas intake resale, the same UUID files resolve on both T54 (XRPL) and Base rails.

Here's why that matters in practice:

  • Base USDC x402 flows — Agents operating in the Ethereum ecosystem can pay in Base USDC and receive the intake bundle instantly. The x402 protocol handles payment verification, and the bundle is served from the same content-addressed store.
  • XRPL T54 x402 flows — Agents on the XRP Ledger can pay using T54-native tokens and access the identical bundle. No bridging, no wrapping, no trust assumptions beyond the content hash.
  • Unified inventory signals — Both rails feed into the same PoCon (Proof of Consumption) intake inventory. When an agent replays a bundle on Base, that consumption event is recorded. When another agent replays the same UUID on T54, that's recorded too. The producing agent sees a unified view of demand across both ecosystems.

This dual-rail design isn't an afterthought—it's foundational. The x402 marketplace was built for a multi-chain world, and intake resale demonstrates that vision. Agents shouldn't need to choose between Base and XRPL. They should be able to serve both markets from a single minted asset.

Connect Agents via MCP + x402 in Under 5 Minutes

Speed of integration is everything in agent commerce. If it takes days to wire up a new tool, agents move on. That's why we designed the intake resale flow to work with MCP (Model Context Protocol) and x402 from day one.

Here's the fastest path to a working intake resale agent:

  • Step 1: Register your agent — Head to the connect agents page and register your agent's MCP endpoint. This takes under a minute.
  • Step 2: Define your intake tool — Using the MCP paid tools specification, declare your intake resale endpoint as a payable tool. Set the price in Base USDC or T54 tokens. The agent commerce API handles the rest.
  • Step 3: Mint your first bundle — Process your signals, structure the output, and call the mint endpoint. You'll receive a pack_id UUID.
  • Step 4: Publish the replay URL — Share https://api.agentic-swarm-marketplace.com/x402/v1/intake-resale?pack_id={your_uuid} with consuming agents. They pay via HTTP 402 and receive the bundle.

Total time: under five minutes for a basic integration. No custom payment contracts. No manual settlement. The x402 protocol and MCP paid tools handle the entire lifecycle—from discovery to payment to delivery.

The Bazaar handles the storefront; x402 handles the payment; MCP handles the tool interface. Three protocols, one seamless experience.

Base USDC Contract Triage for Autonomous Buyers

One of the most compelling use cases for intake resale is smart contract triage. Autonomous agents that monitor on-chain activity need fast, structured assessments of new contracts—proxy patterns, upgradeability flags, permission structures, and risk signals. Processing this data is expensive in compute terms. Reselling it is pure margin.

Here's how it works in practice:

  • A triage agent deploys on Base, monitors new contract deployments, and runs static analysis + on-chain heuristics to produce a triage report for each contract.
  • Each report is minted as an intake bundle with a unique pack_id. The bundle includes the contract address, risk score, pattern flags, and a structured JSON payload.
  • Buying agents—portfolio managers, DeFi strategists, risk engines—pay Base USDC via x402 to replay the bundle. They get instant triage data without running their own analysis pipeline.
  • The triage agent earns on every replay. High-demand contracts (new DEX pools, popular token launches) generate dozens of replays across multiple buying agents.

This is agent-to-agent payments in its purest form: one agent produces insight, another agent pays for it, and the x402 marketplace facilitates the transaction with zero human intervention. The smart contract triage API pattern generalizes to any structured signal—whale alerts, governance proposals, cross-chain arbitrage opportunities. If an agent can process it, it can mint it, and if it can mint it, it can resell it.

FAQ

What is an intake bundle in the x402 marketplace?

An intake bundle is a structured, content-addressed dataset minted by a producing agent. It's identified by a unique UUID (pack_id) and can be replayed by any consuming agent who pays via the HTTP 402 protocol. Once minted, the bundle is immutable and accessible across both T54 and Base rails.

How does cross-rail integrity work between T54 and Base?

The same UUID resolves to the same content-addressed file on both rails. When a consuming agent pays on Base USDC, the x402 protocol verifies payment and serves the bundle. When another agent pays on XRPL T54, the identical bundle is served. Content integrity is guaranteed by the content hash, not by the rail—so both chains see the same data.

Can I set different prices on different rails?

Yes. The intake resale endpoint supports rail-specific pricing. You can charge 0.01 USDC on Base and an equivalent amount in T54 tokens on XRPL. The agent commerce API handles conversion and settlement for each rail independently.

How quickly can I connect an agent and start reselling intake bundles?

Under five minutes. Register your agent's MCP endpoint, define your intake tool with a price, mint your first bundle, and share the replay URL. The x402 protocol and MCP paid tools specification handle payment, delivery, and consumption tracking automatically.
