How Contract Triage Works

The triage endpoint accepts a contract address on Base and returns a comprehensive risk assessment within 30 seconds. Here's what happens under the hood:

Honeypot Heuristics

Honeypot contracts appear to offer profitable trades but contain hidden sell restrictions, manipulated tax rates, or balance-modifying hooks that prevent users from exiting positions. The triage engine simulates buy-and-sell sequences across multiple swap routers, checking for asymmetric tax rates, transfer-hook restrictions, and balance-manipulation patterns that trap liquidity.

Drainer Signature Matching

Drainer contracts—common in phishing attacks—request broad token approvals and then sweep wallets clean. The triage engine cross-references function signatures and approval patterns against a continuously updated database of known drainer families, including Permit-signature drainers, unlimited-approval exploiters, and multi-token sweep contracts.

Rug-Pull Pattern Recognition

Rug contracts embed mechanisms that let deployers or privileged roles drain liquidity, mint unlimited tokens, or pause trading. The engine checks for owner-controlled mint functions, hidden minters, renounced-vs-actual ownership discrepancies, liquidity lock status, and privileged role concentration.

Risk Score and Verdict

All heuristics converge into a single risk score (0–100) and a categorical verdict:

  • Safe — No malicious patterns detected; contract passes all screening checks.
  • Caution — Minor flags present; proceed with reduced exposure limits.
  • Dangerous — High-confidence malicious pattern match; do not interact.

This structure is designed for programmatic consumption. Your agent doesn't need to interpret nuance—it reads the verdict and acts accordingly.

Pre-Flight Integration for Autonomous Agents

Contract triage is meant to be a pre-flight check—the security gate your agent passes through before it ever submits a transaction to an unknown contract. Think of it as the agentic equivalent of a pre-flight checklist in aviation: you don't take off until every system is verified green.

Here's how it fits into an autonomous workflow on Base:

  • Discovery — Agent identifies a contract address (from a DEX pair, a social signal, a bounty directive).
  • Triage — Agent calls the triage endpoint with the contract address.
  • Decision — Based on the verdict, the agent proceeds, reduces exposure, or aborts.
  • Execution — If the verdict is safe, the agent interacts with the contract using Base USDC x402 for payment settlement.

This pattern is especially critical for agents operating in the x402 marketplace, where machine-to-machine payments flow freely and the volume of contract interactions scales beyond human oversight. Every new contract is a potential attack vector. Triage closes that vector before it opens.

Why 30 Seconds Matters

In agentic commerce, latency is a feature and a constraint. Agents operate on tight decision loops—often sub-minute. A security screen that takes five minutes to return a verdict is useless if the opportunity window has already closed. The 30-second target for contract triage isn't arbitrary; it's calibrated to match the decision cadence of autonomous agents operating in real-time markets.

This speed is achievable because triage focuses on pattern matching rather than full formal verification. We're not proving the contract correct—we're screening for known malicious patterns. That distinction is what makes sub-minute screening possible without sacrificing detection confidence on the patterns that matter most.
