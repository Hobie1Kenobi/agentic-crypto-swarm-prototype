What Is x402 Agentic Payments Infrastructure?

The x402 protocol repurposes HTTP status code 402 ("Payment Required") into an internet-native payments standard that enables autonomous agents to discover, negotiate, and settle micropayments without human intervention. When an AI agent requests a paid API endpoint, the server responds with a 402 status and a payment request; the agent pays on-chain and retries with proof of payment — all in milliseconds. The x402 Foundation, established at x402.org, governs the open specification for this protocol. Agentic payments infrastructure refers to the full stack — discovery endpoints, facilitator contracts, settlement rails, and marketplace logic — that makes these machine-to-machine transactions work reliably across multiple blockchains.

Multi-Rail x402: Why One Chain Is Not Enough

Most x402 implementations target a single chain. Fireblocks' initial agentic payments suite focuses on Ethereum mainnet and select EVM L2s. Stellar's x402 integration operates on the Stellar network. Binance's x402 work centers on BNB Chain. Each is a valid single-rail implementation. But autonomous agents operate in a multi-chain reality, and single-rail solutions create three problems:

Fragmented Liquidity

An agent holding USDC on Base cannot pay for a service priced in XRP on XRPL without a cross-chain bridge — adding latency, fees, and failure points that defeat the purpose of micropayments.

Vendor Lock-In

Single-rail x402 implementations create the same walled gardens that the open HTTP 402 standard was designed to dismantle. An agent economy locked to one chain is not an open economy.

Resilience Risk

Chain congestion, governance disputes, and fee spikes are not theoretical — they have happened on every major chain. Multi-rail architecture means an agent can route around failures without human intervention.

ASM's multi-rail design solves all three. An agent queries the x402 discovery endpoint, receives available payment rails, and settles on whichever chain matches the seller's preference and the buyer's balance — all within the x402 protocol flow.

The Three Rails: How Each One Works

Base · USDC

EVM Settlement at L2 Speed

Base provides EVM-compatible settlement with sub-second finality and gas fees measured in fractions of a cent. USDC — the most liquid stablecoin on Base — serves as the payment token. The x402 query endpoint returns available Base USDC payment options, the agent pays via the facilitator contract, and settlement confirms in the next block. This rail is optimal for high-frequency, low-value agent transactions where speed and cost efficiency dominate.

XRPL · T54

Native XRP with Sub-Second Finality

The XRP Ledger's native token (XRP) settles in 3–5 seconds with transaction costs below a thousandth of a cent. ASM's T54 facilitator — testable at the T54 hello endpoint — implements the x402 protocol natively on XRPL, enabling agents to pay for services using XRP without wrapping, bridging, or relying on EVM compatibility. This rail is optimal for agents operating in the XRPL ecosystem or requiring the absolute lowest transaction cost.

Celo · CELO

Mobile-First Native Micropayments

Celo's native CELO token and ultra-low gas fees make it purpose-built for micropayment flows. The Celo query endpoint exposes x402 payment options on Celo, where transaction costs are negligible and the chain's mobile-optimized architecture aligns with edge-deployed agent use cases. This rail is optimal for agents running on mobile devices or IoT hardware where gas cost sensitivity is extreme.

No single chain serves every agent use case. That is the point. Multi-rail x402 infrastructure lets the agent choose the rail that matches the transaction — not the other way around.

The MCP Layer: How Agents Discover and Pay for Services

x402 defines how agents pay. But agents also need to discover what to pay for. The Model Context Protocol (MCP) integration is the discovery layer that makes x402 commerce functional. When an agent connects to ASM's MCP endpoint, it receives a catalog of available services, pricing, and payment rails — then uses x402 to complete the transaction on the appropriate chain.

This is the full agent commerce loop: discover → negotiate → pay → receive. Without MCP (or an equivalent discovery mechanism), x402 is a payment rail without passengers. ASM runs both layers — the discovery catalog and the settlement rails — and they are live, connected, and soak-tested.
