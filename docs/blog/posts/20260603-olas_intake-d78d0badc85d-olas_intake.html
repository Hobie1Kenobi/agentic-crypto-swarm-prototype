Olas Intake Resale: Replayable x402 Bundles Across Base and T54

Olas intake bundles are now minted, replayable, and resellable via x402—same UUID across Base and T54 rails, feeding PoCon inventory signals for autonomous agent commerce.

Olas intake

Agent builders have a data problem. Every autonomous agent ingests signals—market feeds, contract events, triage results—but once consumed, that data evaporates. The computation that produced it is gone. What if intake data could be minted, replayed, and resold as a first-class asset? That's exactly what Olas intake resale delivers, and it's live now on the x402 marketplace.

Intake bundles are minted with a deterministic UUID, priced via HTTP 402, and instantly replayable by any agent that presents the right ?pack_id=uuid parameter. The same UUID resolves identically whether your agent pulls from the XRPL T54 x402 rail or the Base USDC x402 rail. No re-minting, no format conversion, no trust gap. One mint, two rails, infinite replay.

How Minted Intake Bundles Work

An intake bundle is a packaged snapshot of processed agent signals—contract triage outputs, filtered event streams, scored opportunity feeds—sealed at mint time with a UUID. Once minted, the bundle becomes an immutable, replayable data product on the x402 marketplace.

Here's the flow:

  • Mint: An Olas agent processes raw signals and packages the output into an intake bundle. The bundle is assigned a UUID and listed for resale via the agent commerce API.
  • List: The bundle appears on the x402 marketplace with a price denominated in USDC (Base) or XRP (T54). Buyers discover it through standard HTTP 402 negotiation.
  • Replay: A purchasing agent sends a request to /x402/v1/intake-resale?pack_id={uuid}. The server validates payment, then streams the exact bundle payload. Same data, same structure, same deterministic output—every time.

This isn't cached data or a stale snapshot with drift risk. The UUID is a content-addressed commitment. Replay is idempotent. If two agents purchase the same bundle and replay it with the same pack_id, they receive byte-identical results. That's the foundation for composable, trust-minimized agent-to-agent payments.

Cross-Rail Portability: One UUID, Two Chains

Interoperability between chains is usually a nightmare of bridge contracts, wrapped tokens, and settlement delays. Intake bundles sidestep all of it. The bundle payload lives off-chain in the resale layer; the payment rail is the only on-chain component. This means the same UUID file resolves identically on both the T54 (XRPL) and Base rails.

Why does this matter for agent builders?

  • Payment flexibility: An agent funded on Base pays in USDC. An agent funded on XRPL pays in XRP. Both access the same intake bundle. No bridge, no delay, no slippage.
  • Deterministic replay: Regardless of which rail funded the purchase, the replayed payload is identical. Downstream agents can compose on top of intake data without caring about the settlement layer.
  • Inventory coherence: PoCon intake inventory signals aggregate purchases across both rails under the same UUID. Sellers get a unified view of demand; buyers get a unified view of supply.

This dual-rail architecture is what makes intake resale a genuine machine-to-machine commerce primitive. Agents don't need to share a chain to share data. They just need to agree on the UUID.

PoCon Inventory Signals: Closing the Feedback Loop

Every intake resale transaction feeds PoCon intake inventory signals—a real-time feed of what's being bought, how often, and at what price. For sellers, this is demand discovery. For buyers, this is supply discovery. For the network, this is price formation.

PoCon signals enable several powerful patterns:

  • Dynamic pricing: If a smart contract triage API bundle sees high replay volume, the seller can adjust pricing upward. If demand drops, prices adjust down. Market-driven, not admin-set.
  • Bundle composition: Agents can subscribe to PoCon signals to identify which intake types are undersupplied and mint new bundles to fill the gap. This is autonomous market-making for data products.
  • Quality signaling: High replay rates and repeat purchases signal bundle quality. Low replay rates after purchase signal stale or low-value data. The market self-curates.

For builders integrating via MCP paid tools, PoCon signals are accessible as a standard tool output. Your agent doesn't need a separate analytics pipeline—it just calls the intake inventory tool and gets structured signal data back through the same MCP connection.

Connect Agents via MCP + x402 in Under 5 Minutes

Getting intake resale running in your agent stack is straightforward. Here's the minimal integration path:

Step 1: Register Your Agent

Head to the agent connection page and register your agent with the marketplace. You'll receive an agent ID and API credentials. This takes under a minute.

Step 2: Add MCP Tools

Install the intake resale MCP tools into your agent's tool registry. The tools expose two primary actions: list (discover available intake bundles filtered by type, rail, and price) and replay (purchase and stream a bundle by UUID). Both actions handle HTTP 402 payment negotiation automatically.

Step 3: Fund Your Agent's Wallet

For Base USDC x402 purchases, fund your agent's Base wallet with USDC. For T54 purchases, fund with XRP. The marketplace handles payment routing based on the bundle's listed rail.

Step 4: Replay

Call the replay tool with a pack_id. The tool handles the full cycle—payment, retrieval, deserialization—and returns the intake payload to your agent's context window. Your agent can now act on the data immediately.

That's it. No custom payment logic. No rail-specific adapters. No manual UUID management. The x402 marketplace abstracts the entire commerce layer so your agent focuses on reasoning, not settlement.
