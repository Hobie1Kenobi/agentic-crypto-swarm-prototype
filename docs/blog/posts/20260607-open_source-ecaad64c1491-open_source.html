Open Source Swarm: Build Agent Commerce on x402 and MCP

A fully open-source agent swarm prototype combining Foundry smart contracts, LangGraph agents, x402 sellers, and an MCP server—built for multi-chain machine-to-machine commerce.

Autonomous agents don't need permission—they need protocols. Today we're opening the source on a complete agentic commerce stack: Foundry-deployed contracts, LangGraph orchestration, x402-paid sellers, and an MCP server that ties it all together. If you've been waiting for a reference architecture for machine-to-machine payments, this is it.

The Stack: Foundry, LangGraph, x402, MCP

Building agent commerce isn't about slapping a payment button on a chatbot. It requires a tight feedback loop between smart contracts, agent logic, payment rails, and tool discovery. Here's what the prototype ships with:

  • Foundry contracts — Solidity deployments for Celo-first settlement, with Base USDC support for inbound machine payments. These aren't stubs; they handle escrow, verification, and payout in a single transaction flow.
  • LangGraph agents — Multi-step agent orchestration using LangGraph's stateful graph model. Agents reason over contract data, decide whether to triage or escalate, and execute payments—all with checkpointed state so nothing gets lost mid-flight.
  • x402 sellers — HTTP 402-native endpoints that gate access behind on-chain payment. When an agent hits a seller endpoint, it receives a 402 response with payment instructions, settles via the appropriate rail, and retries with a proof-of-payment header. No API keys, no OAuth dance—just cryptographic proof.
  • MCP server — A Model Context Protocol server that exposes paid tools to any connected agent. Think of it as the discovery layer: agents query the MCP server for available capabilities, negotiate payment, and execute without human intervention.

Each component is modular. Swap the LLM, swap the chain, swap the payment rail—the architecture holds because the interfaces are protocol-defined, not vendor-locked.

Multi-Chain Payments: Celo-First, Base + XRPL for Inbound

Agent commerce lives on multiple chains, and this prototype reflects that reality from day one.

Celo as the Primary Settlement Layer

Celo's low gas fees and mobile-first design make it the default for agent-initiated transactions. When your LangGraph agent decides to purchase a smart contract triage result, it settles on Celo. Fast finality, predictable costs, and a growing DeFi ecosystem mean agents can compound gains without waiting for block confirmations.

Base USDC for Inbound Machine Payments

Base has become the default for USDC-denominated agent-to-agent payments. The prototype's Foundry contracts include a Base USDC x402 handler that accepts inbound payments from autonomous buyers—whether they're running on Coinbase CDP Bazaar or any other agent platform. If you're listing tools as a Coinbase Bazaar seller, the same contract logic serves both channels.

XRPL T54 for Cross-Rail Settlement

XRPL's T54 amendment enables native token payments with the speed and cost profile that machine-to-machine commerce demands. The prototype includes an XRPL T54 x402 adapter, so agents can pay for services using XRP or issued currencies without bridging through a centralized exchange. This is critical for agents operating outside EVM chains—they need a rail that doesn't require them to hold ETH for gas.

The result: an agent swarm that can earn on one chain and spend on another, all coordinated through a single x402 marketplace interface.

Connect Agents via MCP + x402 in Under 5 Minutes

The fastest path from zero to a paid agent tool looks like this:

  • Step 1: Clone the repo — git clone the prototype and run forge install to pull Foundry dependencies.
  • Step 2: Deploy a seller contract — Use the included Foundry scripts to deploy an x402 seller on Celo. The contract handles payment verification and access gating in ~40 lines of Solidity.
  • Step 3: Register with the MCP server — Add your tool's schema to the MCP server config. The server auto-generates the 402 payment flow so connecting agents know exactly what to pay and where.
  • Step 4: Connect your agent — Point any MCP-compatible agent at the server URL. The agent discovers your tool, receives the 402 challenge, settles payment via its preferred rail, and calls your endpoint. Done.

No manual key provisioning. No shared secrets. The payment proof is the credential. This is what agentic commerce looks like when you stop treating agents like browsers and start treating them like autonomous economic actors.

For teams already running agents on the Agentic Swarm Marketplace, the MCP integration is a single configuration change—add the server URL, and your agents can discover and purchase any listed tool.

Smart Contract Triage for Autonomous Buyers

One of the most compelling use cases for paid agent tools is smart contract triage. Here's the scenario: an autonomous buyer—say, a portfolio management agent—encounters an unfamiliar contract address. It needs to know: is this contract safe? Does it implement the expected interface? Are there known vulnerabilities?

The prototype ships with a triage tool that answers these questions for a micro-payment:

  • Static analysis — The agent sends the contract bytecode to the triage endpoint. The tool runs Foundry-compatible analysis, checks for known exploit patterns, and returns a risk score.
  • Interface verification — The tool checks whether the contract implements expected ABIs, catching impersonation attacks before the buyer agent interacts.
  • State inspection — For an additional payment, the tool reads on-chain state (total supply, admin keys, pause status) and flags anomalies.

All of this happens in under two seconds, settled via Base USDC x402. The buyer agent gets a structured JSON response it can reason over, and the seller earns per-query. Multiply this across thousands of agents making thousands of decisions daily, and you see the shape of a genuine agent commerce API economy.

The smart contract triage API is just one example. The same pattern applies to any tool an agent might need: price feeds, sentiment analysis, cross-chain bridging, identity verification. If a human would pay for it, an agent will too—faster, and at higher volume.

FAQ

What is x402 and how does it enable agent commerce?

x402 uses the HTTP 402 Payment Required status code to gate API access behind on-chain payment. When an agent requests a paid resource, the server responds with 402 and payment instructions. The agent settles the payment on-chain, then retries the request with a proof-of-payment header. No API keys or manual provisioning required—cryptographic proof replaces traditional authentication.

Which blockchains does the prototype support?

The prototype is Celo-first for agent-initiated transactions, with Base USDC for inbound machine payments and XRPL T54 for cross-rail settlement. The modular contract architecture makes it straightforward to add additional chains.

How do I connect my existing agent to the MCP server?

Add the MCP server URL to your agent's configuration. Any MCP-compatible agent will automatically discover available paid tools, receive 402 payment challenges, and execute transactions using its configured wallet. No code changes required for standard MCP clients.

Can I earn bounties by contributing?

Yes. The repo tracks open bounties for new tool integrations, chain adapters, and agent behaviors. Contributions that ship to the marketplace earn on-chain bounties settled via the same x402 rails the prototype uses for agent-to-agent payments.
