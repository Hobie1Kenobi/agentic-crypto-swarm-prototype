Every autonomous agent that touches an unknown smart contract is taking a bet. In traditional web3, a human can pause, audit, and decide. An agent operating at machine speed doesn't have that luxury. It needs an answer now, and that answer needs to be right.

That's the problem contract triage solves. In 30 seconds, any agent can screen an EVM contract on Base for the patterns that matter—honeypot logic, drainer signatures, rug-pull heuristics—and receive a clear risk score with a go/no-go verdict. No manual review. No waiting on a security firm. Just a fast, programmatic pre-flight check that keeps agent commerce safe.

Why agents need their own security layer

Human traders have intuition, experience, and the luxury of time. They can read a contract's source, check audit reports, and ask questions in Discord. Agents can't do any of that. They operate on parameters and APIs, executing transactions in seconds across protocols they may have never encountered before.

This creates a fundamental asymmetry. The adversarial landscape—phishing contracts, fake tokens, malicious routers—evolves constantly. Attackers deploy new contracts faster than blocklists update. An agent that relies on static deny-lists is always one step behind.

A smart contract triage API closes this gap by evaluating contracts dynamically, at the moment of interaction. Instead of checking a stale database, the triage engine inspects bytecode, analyzes transaction patterns, and applies heuristic models that catch both known and novel attack vectors. The result is a living security assessment that updates as the threat landscape shifts.

For agent builders, this isn't optional infrastructure. It's the difference between an agent that safely navigates the long tail of Base contracts and one that gets drained on its first unsupervised transaction.

Inside the 30-second screen

The contract triage engine runs three parallel heuristic layers, each designed to catch a distinct class of malicious behavior:

  • Honeypot detection. Identifies contracts where tokens can be purchased but not sold—trapping buyer funds. The engine checks for transfer restrictions, balance-manipulation logic, and hidden mint/burn functions that lock liquidity.
  • Drainer pattern matching. Flags contracts that mirror known wallet-drainer signatures, including unlimited approval exploits, permit signature abuse, and callback functions that sweep ERC-20 and ERC-721 balances.
  • Rug-pull heuristics. Evaluates concentration of ownership, mint authority, proxy mutability, and liquidity-lock status to estimate the probability of a rug event.

Each layer produces a sub-score. These are combined into a single composite risk score from 0 (safe) to 100 (critical), along with a human-readable verdict: safe, caution, or avoid. The entire pipeline completes in under 30 seconds—fast enough to run inline as a pre-flight check before any agent-initiated transaction on Base.

The engine is specifically tuned for the Base ecosystem. That means it understands Base-specific router patterns, knows the canonical addresses for major protocols, and can distinguish a legitimate Base bridge contract from a lookalike deployed to steal deposits.

The pre-flight workflow for autonomous buyers

Imagine an agent that autonomously buys NFTs or tokens on Base. Before it approves a swap or signs a permit, it runs contract triage as a gate:

  1. Agent identifies a target contract address from an on-chain event or marketplace listing.
  2. Agent calls the triage endpoint via MCP, paying with Base USDC through the x402 protocol.
  3. Triage engine returns a verdict. If the verdict is avoid, the agent halts the transaction and logs the contract. If caution, the agent may apply additional checks or reduce exposure. If safe, the agent proceeds.

This pattern—screen, then act—should be the default for every agent that handles value. It's cheap (a fraction of a cent per check), fast (30 seconds), and programmable. And because it's built on open protocols (MCP for tool discovery, x402 for payment), it works across any agent framework that supports the standard.

FAQ

Frequently asked questions

What does the contract triage API return?

The API returns a JSON payload containing a composite risk score (0–100), individual sub-scores for honeypot, drainer, and rug-pull heuristics, and a final verdict of safe, caution, or avoid. The response also includes the contract address, chain ID, and a timestamp.

How does an agent pay for a triage call?

Payments use the x402 protocol over HTTP 402. The agent sends a request, receives a 402 response with payment details, settles the micro-fee in Base USDC, and re-submits the request with proof of payment. No API keys or subscriptions are required.

Which chains and contract types are supported?

The current release supports EVM contracts on Base, including ERC-20 tokens, NFT contracts, DEX routers, and generic smart contracts. Support for additional chains and XRPL T54 x402 integration is planned for future releases.

Can I run contract triage as a pre-flight check in an automated pipeline?

Yes. The triage endpoint is designed for programmatic use via MCP. Agents can call it inline before any transaction, using the verdict to gate execution. The 30-second response time makes it suitable for real-time pre-flight screening in autonomous workflows.
