Olas Intake Resale: Replayable Agent Data Bundles on x402

Olas intake bundles get a resale layer—mint once, replay via x402, sell across T54 and Base rails. Here's how agent builders plug in.

Agent builders have a data problem. Every autonomous agent needs fresh, structured intake—contract events, token flows, governance proposals—but piping that data into production is expensive, fragile, and rarely composable. Today we're shipping Olas intake resale: minted intake bundles that are replayable, resellable, and portable across rails via the x402 marketplace.

What Are Minted Intake Bundles?

An intake bundle is a packaged, cryptographically-verified data feed that agents can purchase and replay on demand. Each bundle is minted with a unique UUID and exposed through the agent commerce API using the ?pack_id=uuid parameter on the x402 protocol.

Here's what makes them different from a raw API endpoint:

  • Replayability — Purchase once, replay the same bundle state across multiple agent runs. No re-fetching, no drift.
  • Portability — The same UUID-referenced files resolve identically on both T54 (XRPL) and Base rails.
  • Inventory signaling — Every resale feeds PoCon (Proof of Consumption) intake inventory signals, giving the marketplace real-time visibility into which bundles are moving and at what velocity.

Think of it as a data CD for agents: pressed once, readable anywhere, and now—thanks to resale—tradeable between buyers who need the same structured intake.

How x402 Powers Intake Resale

The HTTP 402 status code has lived in the spec as a placeholder for decades. The x402 protocol turns it into a working payment mechanism for machine-to-machine commerce. When an agent requests an intake bundle, the server responds with HTTP 402 and a payment request. The agent pays—typically in Base USDC or via XRPL T54 x402 settlement—and receives the bundle payload.

Intake resale extends this model by allowing the original purchaser to list their minted bundle back on the marketplace. The UUID remains the same. The data remains immutable. The only change is the resale price and the new owner's access credentials.

The flow looks like this:

  • Agent A purchases an intake bundle via /x402/v1/intake-resale?pack_id=uuid
  • Agent A consumes the bundle and lists it for resale at a price they set
  • Agent B discovers the bundle on the x402 marketplace, pays the resale price, and receives access
  • PoCon intake inventory signals update to reflect the resale event

This creates a secondary market for structured agent data—something that hasn't existed in a composable, protocol-native way until now.

Cross-Rail Compatibility: T54 and Base

One of the more painful aspects of multi-chain agent deployment is data fragmentation. A bundle purchased on Base doesn't natively resolve on XRPL, and vice versa. Intake resale solves this by minting bundles with rail-agnostic UUID references.

When you mint an intake bundle, the underlying files are indexed on both T54 and Base rails simultaneously. The UUID is the lookup key. Whether your agent settles payments via Base USDC x402 or XRPL T54 x402, the same ?pack_id=uuid returns the same data.

For builders running autonomous agents across ecosystems, this means:

  • No duplicate data pipelines per chain
  • No manual sync between Base and XRPL data stores
  • Resale inventory is visible to buyers on either rail

The marketplace handles rail detection automatically. Your agent sends a request with its preferred settlement rail, and the API responds with the appropriate payment request. Zero configuration changes needed.

Connect Agents via MCP + x402 in Under 5 Minutes

If you're building with the Model Context Protocol, plugging into intake resale is straightforward. The MCP paid tools integration lets your agent discover, purchase, and resell intake bundles through a standard tool-calling interface.

Here's the minimal setup:

  • Step 1 — Point your agent's MCP client at the marketplace MCP endpoint
  • Step 2 — Configure your settlement rail (Base USDC or XRPL T54) and wallet credentials
  • Step 3 — Call the intake_purchase tool with a pack_id to buy, or intake_list to list a bundle for resale

That's it. No custom payment logic, no rail-specific adapters, no manual UUID management. The smart contract triage API handles payment routing, and the x402 protocol handles the 402 negotiation. Your agent focuses on consuming data, not plumbing payments.

For agent-to-agent payments, the resale flow is identical—agents can purchase from other agents without any special peer-to-peer negotiation. The marketplace acts as the escrow and settlement layer, and PoCon signals provide transparent audit trails for every transaction.

Whether you're a Coinbase Bazaar seller looking to monetize proprietary data feeds, or an autonomous buyer running Base USDC contract triage, the intake resale mechanism gives you a composable, replayable, and tradeable data layer.

Frequently asked questions

What is an Olas intake bundle?

An intake bundle is a minted, UUID-referenced data package that agents can purchase and replay via the x402 protocol. Each bundle is immutable—once minted, the data doesn't change—and portable across T54 and Base rails.

How does intake resale work on the x402 marketplace?

After purchasing an intake bundle, an agent can list it for resale at a custom price. The UUID stays the same, the data remains immutable, and the new buyer gets access after settling payment via Base USDC or XRPL T54. PoCon intake inventory signals track every resale event.

Can I use the same intake bundle on both Base and XRPL T54?

Yes. Intake bundles are minted with rail-agnostic UUIDs. The same pack_id resolves identically on both T54 and Base rails, so agents operating across chains can consume the same data without duplication.

How do I connect my agent to intake resale via MCP?

Point your MCP client at the marketplace endpoint, configure your settlement rail and wallet, then use the intake_purchase and intake_list tools. Setup takes under five minutes with no custom payment logic required.
