What's Inside the Repository

The prototype is a working reference architecture, not a slide deck. Clone it, deploy contracts, spin up agents, and watch them transact. Here's the breakdown:

  • Foundry contracts — Solidity contracts for Celo-first deployment, with Base and XRPL adapters for inbound machine payments. These handle escrow, settlement, and verification so agents can pay each other trustlessly.
  • LangGraph agents — Multi-step agent workflows built on LangGraph. Each agent can reason over tasks, call paid tools via MCP, and settle invoices through x402 endpoints.
  • x402 sellers — HTTP 402 payment-gated endpoints that agents expose to the marketplace. A buyer agent sends USDC on Base or XRP via T54 on XRPL; the seller agent releases the resource once payment confirms.
  • MCP server — A Model Context Protocol server that registers paid tools, handles authentication, and routes requests between agents. This is the connective tissue that makes MCP paid tools discoverable and callable.

Every component is modular. Swap the LLM, change the chain, add a new tool—the architecture doesn't lock you into a single provider or network.

Connect Agents via MCP + x402 in Under 5 Minutes

The fastest path from zero to a transacting agent looks like this:

  1. Deploy contracts — Run forge script against Celo Alfajores (testnet) or Celo mainnet. The scripts handle proxy setup, initializer encoding, and verification.
  2. Start the MCP server — cd mcp-server && npm run start. The server exposes a smart contract triage API that agents can call to analyze, classify, and route contract interactions.
  3. Register x402 endpoints — Use the provided seller template to wrap any function behind an HTTP 402 paywall. Set your price in Base USDC x402 or XRPL T54 x402, and the middleware handles payment verification before the function executes.
  4. Connect to the marketplace — Point your agent at the marketplace connect endpoint. The agent commerce API discovers your tools, lists them for buyers, and routes incoming requests.

Five minutes, zero custom payment logic. The x402 middleware and MCP server handle the hard parts—payment verification, tool discovery, and request routing—so you can focus on what your agent actually does.

Base USDC Contract Triage for Autonomous Buyers

One of the reference use cases in the repo is smart contract triage. Here's why it matters: autonomous buyers on the x402 marketplace need to evaluate contracts before interacting with them. Is this contract verified? Does it have known vulnerabilities? What's the gas cost profile? Is the issuer reputable?

The triage agent in the repo does exactly this. A buyer agent sends a contract address to the smart contract triage API via MCP. The triage agent:

  • Fetches bytecode and ABI from on-chain registries.
  • Runs static analysis against known vulnerability patterns.
  • Checks verification status and source code availability.
  • Estimates gas costs for common interactions.
  • Returns a structured risk score and recommendation.

The buyer pays for this analysis in Base USDC x402—a few cents per request, settled instantly. No subscriptions, no invoices, no human approval workflows. Machine-to-machine commerce at its most efficient.

This pattern generalizes. Any tool that produces structured, valuable output—price feeds, risk scores, data transforms—can be wrapped as an x402 endpoint and listed on the marketplace. The triage agent is just the first example.

Multi-Chain Payments: Celo, Base, and XRPL

The prototype is Celo-first for contract deployment, but inbound payments flow across three chains:

  • Celo — Primary deployment target. Low gas fees, stable-coin-native design, and fast finality make it ideal for agent escrow and settlement contracts.
  • Base — Base USDC x402 payments for buyers operating in the Coinbase ecosystem. Base's EVM compatibility and Coinbase's CDP integration make it a natural on-ramp for Coinbase Bazaar seller endpoints.
  • XRPL — XRPL T54 x402 payments for buyers who prefer XRP-denominated settlement. The T54 standard enables machine-to-machine payments with sub-second finality and negligible fees.

Agents don't need to care about the underlying chain. The x402 middleware abstracts payment verification—USDC on Base, XRP via T54 on XRPL, or CELO stablecoins all resolve to the same seller experience. The agent commerce API handles routing and conversion where needed.

This multi-chain approach isn't accidental. Agent commerce will be multi-chain by default. Buyers will hold balances wherever it's cheapest and fastest. Sellers will accept whatever the market sends. The prototype reflects that reality from day one.

FAQ

What is x402 and how does it enable agent commerce?

x402 is the HTTP 402 Payment Required status code repurposed for machine-to-machine transactions. When an agent calls an x402 endpoint, the server responds with a payment request (amount, currency, destination). The buyer agent sends payment on-chain, then retries the request with proof of payment. The server verifies on-chain and returns the resource. No API keys, no OAuth, no human-managed billing—just cryptographic proof of payment.

How do MCP paid tools work?

The Model Context Protocol (MCP) lets agents discover and call tools through a standardized interface. In this prototype, the MCP server registers tools that require payment. When an agent calls a paid tool, the MCP server routes the request through x402 middleware. The buyer agent pays, the server verifies, and the tool executes. This makes MCP paid tools as easy to call as free ones—the payment layer is transparent to the agent's reasoning loop.

Why Celo-first with Base and XRPL for payments?

Celo offers low-cost, fast-finality contract deployment with native stablecoin support—ideal for escrow and settlement logic. Base provides deep liquidity for USDC payments and tight integration with Coinbase's developer platform. XRPL with T54 enables sub-second, near-zero-fee payments in XRP. Together, they cover the three most practical rails for agent-to-agent payments in 2025.
