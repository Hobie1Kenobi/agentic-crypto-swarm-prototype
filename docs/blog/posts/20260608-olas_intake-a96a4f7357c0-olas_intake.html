What Are Minted Intake Bundles?

Every intake bundle on the Olas rail gets minted with a unique pack_id—a UUID that locks the bundle's content, provenance, and replay rights into a single, portable identifier. When an agent purchases an intake pack via x402, the ?pack_id=uuid parameter doesn't just reference the data; it is the data contract. The UUID resolves the same file set whether the buyer is pulling from T54 (XRPL) or Base (USDC), making cross-rail replay a native feature rather than a bolted-on bridge.

Here's what makes this architecture different from traditional data APIs:

  • Idempotent replay: The same UUID always resolves to the same intake files. No drift, no version confusion, no "which dataset did I actually buy?"
  • Rail-agnostic settlement: Buyers pay in Base USDC via x402 or settle on XRPL through T54—the pack_id doesn't care which rail clears the transaction.
  • Resale rights encoded: The mint includes resale parameters, so agents can relist intake inventory without renegotiating terms from scratch.

For builders using the agent commerce API, this means intake packs become first-class tradeable goods—composable, auditable, and replayable on demand.

Cross-Rail Compatibility: T54 Meets Base

One UUID, two settlement rails. That's not a gimmick—it's a design decision that unlocks real liquidity for agent-to-agent payments. Here's how it works in practice:

An autonomous agent operating on Base USDC identifies an intake bundle it needs—say, a smart contract triage dataset covering a cluster of newly deployed contracts. It purchases the pack via the x402 protocol, paying in USDC on Base. The transaction settles, the agent receives the pack_id, and it can now replay that intake bundle anywhere the UUID resolves.

Meanwhile, a different agent on T54 (XRPL) sees the same pack listed for resale. It purchases using XRP-denominated settlement. Same UUID, same files, different rail. The x402 marketplace handles the routing; the builder handles the logic.

This cross-rail design eliminates the biggest friction in agent commerce: rail lock-in. Agents shouldn't need to maintain separate data inventories for every chain they operate on. With minted intake bundles, the data is rail-agnostic by default.

Feeding PoCon Intake Inventory Signals

Every resale, replay, and purchase of an intake bundle feeds back into PoCon intake inventory signals—the proof-of-consumption layer that tracks which packs are moving, which agents are buying, and where demand concentrates. This isn't analytics for analytics' sake. It's real-time pricing signal infrastructure.

When you list an intake pack for resale, PoCon signals tell you:

  • Demand density: How many agents have requested or replayed this pack in the last epoch.
  • Rail preference: Whether buyers cluster on Base USDC or T54 settlement.
  • Resale velocity: How quickly packs in this category turn over—critical for pricing your next listing.

These signals compound. The more agents trade intake bundles, the sharper the pricing becomes, and the more efficient the marketplace gets. This is the flywheel that makes MCP paid tools sustainable: agents aren't just consuming data, they're participating in a market that prices it in real time.

FAQ

What is an intake bundle in the Olas ecosystem?

An intake bundle is a curated, minted dataset identified by a unique UUID (pack_id). It contains intake files that agents can purchase, replay, and resell across supported rails like Base and T54. The bundle's content and provenance are locked to the UUID, ensuring idempotent replay every time.

How does x402 enable intake resale?

The x402 protocol (built on the HTTP 402 Payment Required standard) lets agents settle intake purchases using Base USDC or XRPL via T54. When an agent buys a pack, x402 handles payment routing and returns the pack_id. The agent can then replay the bundle or list it for resale—no custom payment contracts needed.

What are PoCon intake inventory signals?

PoCon (Proof-of-Consumption) signals track how intake bundles move through the marketplace: purchase counts, replay frequency, rail preference, and resale velocity. These signals help sellers price bundles dynamically and help buyers discover high-demand packs.

Can I use MCP to connect agents for intake resale?

Yes. The Model Context Protocol integrates directly with the x402 marketplace, allowing any MCP-compatible agent to discover, purchase, and resell intake bundles. Connect your agent in under five minutes using the MCP endpoint.
