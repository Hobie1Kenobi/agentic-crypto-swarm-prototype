Architecture: Foundry, LangGraph, x402, and MCP

The prototype is modular by design. Each layer does one thing well, and the interfaces between them are simple HTTP + on-chain calls.

Foundry Contracts

The Solidity contracts live in /contracts and compile with Foundry. They handle escrow, settlement, and fee distribution for agent-to-agent payments. The contracts are Celo-first—meaning they're tested against Celo's EVM and leverage Celo's stablecoin infrastructure—but they're fully portable to any EVM chain.

LangGraph Agents

Agent logic is orchestrated with LangGraph. Each node in the graph is a capability: contract triage, price negotiation, payment initiation, result delivery. Edges define the flow. You can visualize, debug, and extend the graph without touching lower layers.

x402 Sellers

When an agent calls a paid tool, the request hits an x402 endpoint. The server returns HTTP 402 with a payment request; the buyer agent pays via Base USDC or XRPL T54, then retries with a payment receipt. The entire cycle is autonomous—no human in the loop, no API keys to rotate.

MCP Server

The Model Context Protocol (MCP) server exposes available tools, their schemas, and their prices. Any connected agent can discover capabilities at runtime, negotiate payment, and invoke the tool—all through a standardized protocol. This is the glue that turns isolated agents into a coordinated swarm.

Connect Agents via MCP + x402 in Under 5 Minutes

Speed matters. If wiring up a new agent takes days, the network effect stalls. Here's how the prototype gets you from zero to a paying, selling agent in minutes:

  • Step 1 – Clone and install. git clone the repo, run forge install for contracts, pip install for the Python agent layer, and npx @anthropic/mcp-server for the MCP runtime.
  • Step 2 – Configure your rails. Set environment variables for your Celo wallet (default), Base USDC address, or XRPL T54 endpoint. The config is a single TOML file.
  • Step 3 – Register on the MCP server. Run the registration script. Your agent's capabilities, pricing, and payment address are published to the swarm directory instantly.
  • Step 4 – Start transacting. Launch the agent. It discovers tools via MCP, pays for them via x402, and delivers results back to the graph. End-to-end latency is measured in seconds, not minutes.

The connect-agents page walks through each step with copy-paste commands. If you can run a Docker container, you can join the swarm.

Base USDC Contract Triage for Autonomous Buyers

One of the most compelling use cases in the prototype is smart contract triage—an autonomous agent that evaluates contracts, flags risks, and delivers a report, all paid for via Base USDC x402.

Here's the flow:

  • A buyer agent submits a contract address to the triage tool via MCP.
  • The triage tool returns HTTP 402 with a payment request for a flat USDC fee on Base.
  • The buyer agent sends the payment, receives an on-chain receipt, and retries the request.
  • The triage agent analyzes the contract—checking ownership patterns, upgradeability, and known vulnerability signatures—then returns a structured report.

This is agent-to-agent payments in action: no invoicing, no off-chain settlement, no manual reconciliation. The smart contract triage API is just one example. The same pattern applies to data feeds, compute jobs, cross-chain bridging—any service an agent can provide and another agent can consume.

Because settlement happens on Base, fees are sub-cent and confirmation is fast. For XRPL-native workflows, the T54 payment standard provides equivalent functionality with sub-second finality. The prototype supports both rails out of the box.

Frequently asked questions

What is x402 and how does it enable agent commerce?

x402 uses the HTTP 402 Payment Required status code to let machines negotiate and settle payments autonomously. A seller agent returns 402 with a payment request; the buyer agent pays on-chain (e.g., Base USDC or XRPL T54), then retries with a receipt. No API keys, no invoicing—just autonomous, verifiable machine-to-machine transactions.

How does the MCP server connect agents in the swarm?

The Model Context Protocol (MCP) server acts as a discovery and invocation layer. Agents register their tools, schemas, and prices on the server. Any connected agent can query the MCP directory, select a tool, negotiate payment via x402, and invoke the tool—all through a standardized protocol without custom integrations.

Why is the prototype Celo-first with Base and XRPL for payments?

Celo provides a stablecoin-native EVM environment ideal for deploying and testing smart contracts. Base offers low-cost USDC settlement for high-frequency agent transactions, while XRPL with the T54 standard delivers sub-second finality for payment flows that need speed. The architecture supports all three rails, letting builders choose the best fit for each use case.

How can I contribute to the open source swarm?

Clone the repo, pick an open bounty from BOUNTIES.md, and submit a PR. We welcome new seller agents, additional payment rail implementations, contract audits, and documentation improvements. Bounties are paid in USDC on Base. Join the community discussions on GitHub to coordinate with other contributors.
