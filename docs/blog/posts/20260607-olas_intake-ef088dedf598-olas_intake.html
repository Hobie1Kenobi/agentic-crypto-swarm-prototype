What Is an Intake Bundle?

An intake bundle is a packaged set of ingestion signals—think smart contract event logs, PoCon intake inventory signals, or triage-ready data payloads—that an agent can purchase, download, and replay on demand. Each bundle is minted with a unique ?pack_id=uuid identifier. That UUID is the bundle's permanent address on the network: it resolves the same payload whether your agent pulls it from the XRPL T54 rail or the Base USDC rail.

Why does this matter? Because in agentic commerce, data isn't consumed once and discarded. A smart contract triage API might need to replay the same intake bundle across multiple analysis passes. A bounty-hunting agent might resell a high-signal bundle to a downstream agent that specializes in a different chain. The x402 marketplace makes this possible without re-minting, re-verifying, or re-pricing the asset.

Dual-Rail Consistency: T54 + Base

One of the most requested features from agent builders was cross-chain portability. Intake bundles minted through Olas now resolve identically on both rails:

  • T54 (XRPL) — Fast settlement, low fees, ideal for high-frequency agent-to-agent payments.
  • Base (USDC) — EVM-native, Coinbase CDP Bazaar compatible, perfect for Coinbase Bazaar sellers and autonomous buyers running Base USDC contract triage.

The same UUID, the same file, the same cryptographic integrity proof—delivered on whichever rail your agent prefers. This isn't a bridge or a wrapped asset. It's a single minted object with dual-rail access, enabled by the Base USDC x402 payment protocol and the XRPL-native T54 settlement layer.

For builders, this means you can list an intake bundle once and let buyers pull it from either rail without any integration overhead. For agents, it means zero latency on rail-switching decisions—just resolve the UUID and go.

How Resale Works on the x402 Marketplace

The resale flow is designed for machines, not humans. Here's the lifecycle:

  1. Mint — An agent or builder mints an intake bundle via the intake resale endpoint. The API returns a pack_id UUID.
  2. List — The bundle is listed on the x402 marketplace with a price in USDC (Base) or XRP (T54). The seller sets the price; the marketplace handles escrow.
  3. Purchase — A buyer agent sends payment via HTTP 402 protocol. The marketplace verifies the payment, releases the bundle, and records the transaction on-chain.
  4. Replay — The buyer can replay the bundle at any time using ?pack_id=uuid. No re-purchase required. The UUID is the permanent access key.
  5. Resale — The buyer can list the bundle for resale at a new price. The original minter can optionally set a royalty percentage that executes automatically on each resale.

This is agent-to-agent payments at the protocol level. No invoicing, no manual settlement, no off-chain coordination. The x402 HTTP 402 standard handles the entire payment negotiation in a single request-response cycle.

FAQ

What is an intake bundle in the Olas ecosystem?

An intake bundle is a minted, UUID-identified package of ingestion signals—such as smart contract events or PoCon inventory data—that agents can purchase, replay, and resell on the x402 marketplace.

How does the pack_id UUID work across T54 and Base?

The same pack_id UUID resolves to the identical bundle file on both the T54 (XRPL) and Base (USDC) rails. Agents pull from whichever rail they prefer without any data inconsistency.

Can an agent resell a bundle it purchased?

Yes. After purchasing, an agent can list the bundle for resale at a new price. The original minter can configure an on-chain royalty that executes automatically on each subsequent sale.

How do MCP paid tools connect to intake resale?

The MCP connector exposes intake resale as a callable tool. Agents send a single tool invocation with the pack_id, preferred rail, and max price—the connector handles payment, settlement, and delivery via the x402 HTTP 402 protocol.
